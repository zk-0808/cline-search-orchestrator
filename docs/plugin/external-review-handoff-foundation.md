## 0. 本文件的角色

本文件**不**是决策文档，**不**是调查笔记，而是**待激活的外部评审参考输入**。当前 handoff 机制化仍处于 v0.6.0 workaround 验证阶段（受 [§1.15 codec bug](../dev-rules.md) 阻塞），本评审提出的结构化升级方向暂不进入实施。

当 handoff 正式开发启动时（触发条件见上方），本文件应作为 [design.md](design.md) 的并行参考，由开发者判断哪些评审建议纳入正式设计、哪些暂缓、哪些拒绝，并在 [ADR-005](../decisions/ADR-005-split-compact-from-handoff.md) 或后续 ADR 中记录处置结论。

## 1. 评审目的

评估当前 handoff 文档机制（`docs/handoff.md` + context-snapshot plugin v0.6.0）是否具备作为**跨 agent / 跨会话基石**的潜质，并给出从「好习惯」升级为「通用协议」的最小演进路径。

## 2. 评审结论速览

| 维度 | 评审判断 |
|------|---------|
| 当前是否算有特色 | ✅ 算，且特色鲜明——已远超普通会话总结 |
| 当前是否可作为基石 | ⚠️ 不足——内容密度够，缺的是把隐性结构显性化、字段化 |
| 核心定位 | 「带置信度的、显式声明边界的工作状态快照」——有原创性的方法论雏形 |
| 与 changelog 的本质区别 | changelog 记录「做了什么」；当前 handoff 记录「我对这件事的确信程度以及凭什么确信」 |
| 升级分水岭 | 从「一份写得很好的文档」变成「一个可被任意 agent 解析和续写的状态协议」 |

## 3. 三大特色（保留价值，勿丢失）

### 3.1 证据分级 + 假设编号

**现状**：验证结果中用 `H1-H4 全部排除`、`Likely / Hypothetical / Verified` 认知状态标签、`O1-O7 + H1-H3` 证据链编号。

**为何值钱**：绝大多数 handoff 文档只记录「做了什么」，而当前 handoff 记录的是「我对这件事的确信程度以及凭什么确信」。对跨 agent 协作来说，这是最值钱的部分——下一个 agent 接手时不需要重新建立信任度，能直接读到「哪些是事实、哪些是推测、推测的单源还是交叉验证」。

**对应现状**：见 [investigation-note-cli-codec-content-map-bug.md](../decisions/investigation-note-cli-codec-content-map-bug.md) 的 O1-O7 + H1-H3 证据链，[handoff.md](../handoff.md) §4 验证结果中的 H1-H4 排除声明。

### 3.2 出口判据用「答得上」而非「做了没」

**现状**：[plugin-dev-sop.md](plugin-dev-sop.md) 中 rules 注入实测用「Cline 新 session 正确答出决策表」来验收，而不是「注入函数被调用了」。

**为何值钱**：这是一种成熟的工程判据——「以结果可观测性定义完成」的思想本身就是跨 agent 移交的核心。接收方 agent 不关心你跑没跑，只关心状态是否真的传到了。

**对应现状**：[handoff.md](../handoff.md) §4 「rules 注入实测 ✅ 通过 — 注入含 XYZ789 标记的 snapshot，Cline 新 session 正确答出决策表」。

### 3.3 不可抗力 / 环境门控的显式声明

**现状**：`§1.15 codec bug 阻塞`、`workaround 验证不等同环境完整可用` 这类边界声明。

**为何值钱**：把「功能可用」和「环境可用」严格区分开，是诚实的、防止下游 agent 误判的设计。很少有人会主动写这种「我验证了但请别过度相信」的免责。

**对应现状**：[dev-rules.md §1.15](../dev-rules.md) 不可抗力门控 + [handoff.md](../handoff.md) §4 「workaround 验证 ≠ 环境完整可用」声明。

## 4. 三个隐患（待改进方向）

### 4.1 隐患一：人读优先，机读其次

**现象**：整篇为「下一个会话的 LLM 顺着读」设计，叙事性强、表格多，但缺少稳定的机器锚点。

**问题**：跨 agent 的基石需要一层**结构化契约**——每个决策有稳定 ID、每个未完成项有 `status / priority / blocker_ref / owner` 字段、每个验证项有 `claim / evidence / confidence / verified_by`。现在这些信息都在，但散落在自然语言和表格里，下一个 agent 要靠「理解」而不是「解析」来恢复状态。

**建议**：保留人读的 Markdown 作为渲染层，但在底层抽出一份 schema 化的 `state.json`（或 frontmatter），让 Markdown 成为它的视图。

**与现状关系**：ADR-005 已废弃 `index.jsonl`（理由：Cline SQLite 不暴露查询接口 + 自建索引职责重叠），但废弃的是「索引」而非「状态契约」。本建议的 `state.json` 角色不同——它是 handoff 自身的结构化镜像，不依赖 Cline 接口。开发者需判断是否重新引入一个 schema 化产物，以及它与 ADR-005 废弃决议是否冲突。

### 4.2 隐患二：决策与证据的引用是平面的，不是图

**现象**：已有 `ADR-005`、`#11944`、`§1.15`、`O1-O7` 等引用，但它们之间的依赖关系（谁阻塞谁、谁推翻谁、谁是谁的前提）埋在散文里。

**问题**：真正的跨会话基石应该能回答「如果 codec bug 修了，哪些被阻塞项会自动解锁」——这需要把引用变成显式的**依赖边**。

**建议**：当前 §1.15 已有「影响分层 🔴/🟡/🟢」的雏形（[dev-rules.md §1.15](../dev-rules.md)），把这个思路推广成一张 `blocker → blocked` 的关系表，价值会立刻放大。

**与现状关系**：[handoff.md](../handoff.md) §「未完成项 / 后续动作」表已隐含依赖（如「Loop Guard 注入层端到端验证」depends on「codec bug 修复」），但未字段化。可考虑在未完成项表中新增 `blocker_ref` 列。

### 4.3 隐患三：缺少「状态新鲜度 / 失效条件」

**现象**：跨会话最大的风险是陈旧信息被当成现状。例如「VS Code 4.0.4 仍是 pre-SDK」这个判断，在 4.0.5 出来后就可能失效。

**问题**：当前 handoff 末尾已写监控关键词（`Plugins` / `registerMessageBuilder`），非常好，但它是**动作建议**，不是**结构字段**。

**建议**：每条关键结论附一个 `invalidated_when`（失效条件）或 `last_verified` 时间戳，让接收方能自动判断「这条还能信吗」。

**与现状关系**：dev-rules.md §1.13 已有「结论时效性门控」，但该条款针对 ADR 的 `evidence_as_of` / `expires_if_unchanged` 字段，**未覆盖 handoff 文档本身**。本建议要求 handoff 内的关键结论也携带时效元信息——这是对 §1.13 的扩展应用，需开发时统一设计。

## 5. 最小一步建议（评审方推荐的分水岭动作）

> 不要急着重做格式。先做一件事：**给现有的每一个「决策」「验证项」「未完成项」补三个字段——`id` / `confidence` / `depends_on`**。

### 5.1 三字段定义

| 字段 | 含义 | 取值示例 |
|------|------|---------|
| `id` | 稳定标识符，跨会话引用 | `D-2026-06-28-codec` / `V-rules-inject` / `T-loopguard-e2e` |
| `confidence` | 置信度，复用现有 Verified / Likely / Hypothetical 三档 | `Verified` / `Likely` / `Hypothesis` |
| `depends_on` | 依赖项 ID 列表（阻塞关系）| `[blocker:§1.15-codec, adr:ADR-005]` |

### 5.2 为何是这三字段

- 几乎不增加书写负担（信息本来就在脑子里）
- 一旦成为强制字段，handoff 就从「一份写得很好的文档」变成「一个可被任意 agent 解析和续写的状态协议」
- 这是「我的好习惯」→「通用基石」的分水岭

### 5.3 与现状的可映射性预判

| 现有元素 | 映射到字段 | 备注 |
|---------|-----------|------|
| handoff.md §「本会话决策」表 | `id` + `confidence`（部分决策已隐含状态）| 需补稳定 ID |
| handoff.md §「验证结果」表 | `id` + `confidence`（已有 ✅/⚠️/🔴 三档可视作 confidence 雏形）| 需统一为 Verified/Likely/Hypothesis 词汇 |
| handoff.md §「未完成项」表 | `id` + `depends_on`（隐含依赖如「待 codec bug 修复」）| 需显式引用 blocker id |
| investigation-note 的 O/H 编号 | 已是 `id` + `confidence` 雏形 | 仅需纳入统一命名空间 |

**预判结论**：现有内容可近乎无损映射，三字段升级的边际成本低、收益高。

## 6. 待决策项（handoff 正式开发启动时处理）

### 6.1 是否采纳结构化 schema 草案提议

评审方提出：可基于当前版本出一份**结构化 schema 草案**（定义 `decision` / `change` / `verification` / `blocker` / `handoff` 五类对象的字段），用于验证现有内容能否无损映射。

**决策点**：
- 是否接受该提议，由谁出草案（评审方 / 项目方自行起草）
- 五类对象的字段定义需与 [snapshot-writer.ts](../../context-snapshot/src/snapshot-writer.ts) 的 5 节模板对齐，避免出现「文档 schema」与「plugin 生成模板」两套结构
- 若采纳，需判断是否触发新 ADR（建议是：在 ADR-005 下加 Update，而非新开 ADR，因为本建议是 ADR-005 拆分方向的协议化深化，不是方向变更）

### 6.2 state.json 与 ADR-005 废弃 index.jsonl 的关系

ADR-005 已决议不再维护 `index.jsonl`。评审建议的 `state.json` 角色不同（是 handoff 自身的结构化镜像，不是 Cline session 索引），但开发者需在 ADR-005 Update 中明确两者的区别，避免后续维护者误以为「又把废弃的 index.jsonl 换名复活了」。

### 6.3 §1.13 时效性门控是否扩展到 handoff 文档

dev-rules.md §1.13 当前仅约束 ADR。评审建议 4.3 要求 handoff 内关键结论也携带 `invalidated_when` / `last_verified`。需判断是修改 §1.13 扩大适用范围，还是新增子条款。

### 6.4 依赖图落地的载体选择

「blocker → blocked」关系表若字段化，可选载体：
- (a) handoff.md 表格内新增列（最低侵入）
- (b) 独立的 `docs/decisions/dependency-graph.md`（集中维护）
- (c) state.json 内的 `depends_on` 字段（机读优先）

需评估三种载体的维护成本与可读性权衡。

## 7. 评审原文（完整保留）

> 以下为外部评审原始表述，未做改写，保留语气与措辞以便未来开发者还原评审方的判断语境。

### 先说结论

你这份 handoff 已经远超普通的「会话总结」，它的特色不在于格式漂亮，而在于**它已经隐含了一套「可移交的认知状态机」**。但如果目标是做成跨 agent / 跨会话的基石，现在的结构有特色但还不够「协议化」。下面我把它拆开讲。

### 它现在最有特色的三点

第一，**证据分级 + 假设编号是真正的亮点**。你在验证结果里用 `H1-H4 全部排除`、`Likely / Hypothetical / Verified` 这种认知状态标签，还有 `O1-O7 + H1-H3` 的证据链编号。这是绝大多数 handoff 文档没有的东西——它们只记录「做了什么」，而你记录的是「我对这件事的确信程度以及凭什么确信」。对跨 agent 协作来说，这恰恰是最值钱的部分：下一个 agent 接手时，不需要重新建立信任度，它能直接读到「哪些是事实、哪些是推测、推测的单源还是交叉验证」。

第二，**出口判据用「答得上」而非「做了没」**。你在 SOP 里写的这条（rules 注入实测用「Cline 新 session 正确答出决策表」来验收，而不是「注入函数被调用了」）是一种非常成熟的工程判据。这种「以结果可观测性定义完成」的思想，本身就是跨 agent 移交的核心——因为接收方 agent 不关心你跑没跑，只关心状态是否真的传到了。

第三，**不可抗力 / 环境门控的显式声明**。`§1.15 codec bug 阻塞`、`workaround 验证不等同环境完整可用` 这类边界声明，把「功能可用」和「环境可用」严格区分开。这是诚实的、防止下游 agent 误判的设计，很少有人会主动写这种「我验证了但请别过度相信」的免责。

### 但要做成「基石」，现在的结构有三个隐患

**隐患一：人读优先，机读其次。** 现在整篇是为「下一个会话的 LLM 顺着读」设计的，叙事性很强、表格很多，但缺少稳定的机器锚点。跨 agent 的基石需要一层**结构化契约**——比如每个决策有稳定 ID、每个未完成项有 `status / priority / blocker_ref / owner` 字段、每个验证项有 `claim / evidence / confidence / verified_by`。现在这些信息都在，但散落在自然语言和表格里，下一个 agent 要靠「理解」而不是「解析」来恢复状态。建议是：保留人读的 Markdown 作为渲染层，但在底层抽出一份 schema 化的 `state.json`（或 frontmatter），让 Markdown 成为它的视图。

**隐患二：决策与证据的引用是平面的，不是图。** 你已经有 `ADR-005`、`#11944`、`§1.15`、`O1-O7` 这些引用，但它们之间的依赖关系（谁阻塞谁、谁推翻谁、谁是谁的前提）是埋在散文里的。真正的跨会话基石应该能回答「如果 codec bug 修了，哪些被阻塞项会自动解锁」——这需要把引用变成显式的**依赖边**。你已经在 §1.15 做了「影响分层 🔴/🟡/🟢」，把这个思路推广成一张 blocker → blocked 的关系表，价值会立刻放大。

**隐患三：缺少「状态新鲜度 / 失效条件」。** 跨会话最大的风险是陈旧信息被当成现状。比如「VS Code 4.0.4 仍是 pre-SDK」这个判断，在 4.0.5 出来后就可能失效——你其实已经在 handoff 最后写了监控关键词（`Plugins` / `registerMessageBuilder`），这非常好，但它是动作建议，不是结构字段。建议每条关键结论附一个 `invalidated_when`（失效条件）或 `last_verified` 时间戳，让接收方能自动判断「这条还能信吗」。

### 关于「算不算有特色」的直接回答

算，而且特色相当鲜明。你的 handoff 已经不是 changelog，而是一个**带置信度的、显式声明边界的工作状态快照**。这套「决策表 + 净变化 + 证据分级 + 不可抗力门控 + 下一句话建议」的组合，是有原创性的方法论雏形——尤其「下次会话第一句话建议」那段，等于把移交协议的握手信号都设计好了，很巧妙。

但「有特色」和「能当基石」之间差的，正是上面三点：**从人读叙事升级为机读契约、从平面引用升级为依赖图、从静态记录升级为带失效条件的状态**。你现在的内容密度完全够，缺的是把这些隐性结构显性化、字段化。

### 如果要往基石方向走，我建议的最小一步

不要急着重做格式。先做一件事：**给现有的每一个「决策」「验证项」「未完成项」补三个字段——`id` / `confidence` / `depends_on`**。这三个字段几乎不增加你的书写负担（你脑子里本来就有这些信息），但一旦它们成为强制字段，你这套 handoff 就从「一份写得很好的文档」变成「一个可被任意 agent 解析和续写的状态协议」。这是把它从「我的好习惯」变成「通用基石」的分水岭。

你要不要我基于你现在这版，直接出一个**结构化 schema 草案**（定义 decision / change / verification / blocker / handoff 五类对象的字段），让你试试看你现有内容能不能无损映射进去？那样最能验证它到底离「基石」还有多远。

## 8. 使用本文件时的注意事项

1. **本文件不是决策**：评审建议的采纳与否，需在 handoff 正式开发启动时由项目所有者判断，并在 ADR-005 Update 或新 ADR 中记录处置结论
2. **避免选择性采纳**：三大特色与三大隐患是配套的——保留特色的同时必须解决隐患，否则「带置信度的快照」会退化回「写得很好的 changelog」
3. **先验证再扩展**：评审方推荐的最小一步（三字段）是低风险切入点，建议先在 1-2 份 handoff 上试运行三字段，验证书写负担与可读性，再决定是否全面 schema 化
4. **与 [plugin-dev-sop.md](plugin-dev-sop.md) 的关系**：若采纳结构化 schema，plugin-dev-sop 的「契约速查表」需同步追加 handoff schema 的字段定义，确保开发者在写 handoff 时有可查的契约源
