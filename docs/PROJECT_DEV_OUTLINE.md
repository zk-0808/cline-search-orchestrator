# 项目开发提纲：Cline 扩展项目

> 本提纲基于会话压缩摘要提炼，核心目标是**纠正"设计能力 > 调研能力"的失败模式**，建立"先调研再设计、先验证再实现、先决策再推进"的工程纪律，防止工具改造再次演变为无限治理项目。

---

## 0. 写在前面：这个项目到底是什么

### 0.1 定位

```
不是：做一个 Agent 框架
而是：把顺手的工具（Cline）变得更顺手
```

本项目是 **Cline 之上的扩展层**，不是独立的 Agent 系统。Cline 已是相当完整的宿主（Plan/Act、Hooks、Memory Bank、Checkpoint、Rules、Subagents、Workflow、CLI），本项目只补 Cline 真正缺失的部分。

> 项目定位的形态级决策见 [ADR-002](decisions/ADR-002-project-shape.md)：**面向 Cline 用户的开箱即用工作流增强配置——沉淀真实使用经验，并逐步将经验机制化，持续改善长期使用体验**。
>
> 三层定位：**L1 开箱体验**（薄 Skills + 默认规则 + WebSearch MCP）/ **L2 经验沉淀**（项目护城河）/ **L3 经验机制化**（Plugin 作为实验线，**当前不进默认交付**——VS Code Cline 不支持 Plugin）。
>
> 经验机制化清单见 [mechanism-candidates.md](./mechanism-candidates.md)。Plugin 开发参考见 [plugin-dev-quick-reference.md](plugin/refs/plugin-dev-quick-reference.md)。

### 0.2 三个绝不

| 绝不 | 原因 |
|------|------|
| **绝不重复造 Cline 已有能力** | 会导致"设计能力 > 调研能力"的失败重演 |
| **绝不用规则解决运行时问题** | 提示词无法根治 PowerShell 阻塞、session 污染等控制层缺陷 |
| **绝不让战略决断伪装成待办** | 会形成"战略债"，决策不断推迟而执行不断继续 |

## 1. A/B/C 分类纪律（贯穿全程）

**这是项目的最高准则**：遇到任何问题，先分类，再决定处理方式。

### A 类：未来可代码解决的运行时问题

| 特征 | 处理方式 |
|------|----------|
| timeout / watchdog / loop guard / 进程隔离 / UTF8 注入 / 非交互执行 | 若未来掌控执行层，用**机制**替代规则 |
| shell 卡死、session 污染、重复循环、工具异常 | 归入"待机制化"清单，**当前不写规则**，只记录 |

> 原则：A 类问题的正确归宿是代码机制，不是 SKILL.md 里的经验段。现在写规则 = 制造未来要退休的债。

### B 类：当前 Cline 限制导致的运行时问题

| 特征 | 处理方式 |
|------|----------|
| Cline 内核无法修改，但可从外部缓解 | 用 **Hook / MCP / 外部脚本 / Workflow / safe-exec** 打临时补丁 |
| 仅限 Windows Hook 不支持等硬伤 | 打补丁，并标记"Cline 原生支持后立即退休" |

> 原则：B 类是**临时补丁**，必须打"退休标记"，禁止把它当成长期资产精心雕琢。

### C 类：真正的认知问题

| 特征 | 处理方式 |
|------|----------|
| 证据优于推测、问题定义优于方案、复杂度必须证明 | **长期保留**，写进 constitution / 铁律 |
| 时间锚混淆、自我指涉陷阱、假设池遗漏、过快相信首个解释 | 永远不会被代码取代，因为治理的是**思考方式** |

> 原则：C 类才是本项目的**真正长期资产**。精力应集中在这里，而不是 A/B 类。

### 分类决策树

```
遇到问题
  │
  ├─ 是运行时现象（卡死/循环/异常/阻塞）？
  │    │
  │    ├─ 未来能改执行层吗？
  │    │    ├─ 能 → A 类 → 记入"待机制化"，不写规则
  │    │    └─ 不能（Cline 限制）→ B 类 → 打补丁 + 退休标记
  │    │
  │    └─ 否，是思考方式问题 → C 类 → 写进铁律
  │
  └─ 是"缺某能力"？
       │
       └─ 走第 3 节"五问门控"
```

## 2. 停止条件（防止无限优化）

### 2.1 额度触发的强制停止

```
模型额度用完
  → 立即停止开发
  → 进入复盘
```

**目的不是省额度**，而是：
- 限制持续优化冲动
- 强迫阶段总结
- 防止"无限优化 / 无限治理 / 无限重构"

### 2.2 三种必须停止的信号

| 信号 | 含义 | 动作 |
|------|------|------|
| 连续 3 次提交都是"写规则/写记忆/写治理" | 又陷入治理循环 | 停下，回看 A/B/C 分类 |
| 开始写"未来再研究"类待办 | 战略债在累积 | 停下，做决策或明确删除 |
| 出现"我到底在做什么"的困惑 | 定位错位 | 停下，回到第 0 节定位 |

### 2.3 停止后做什么

1. 回顾本周新增了多少规则 vs 推进了多少功能
2. 把新规则过一遍 A/B/C 分类，A/B 类移出长期记忆
3. 明确下一个**功能**目标（不是治理目标）

## 3. 新功能启动门控（五问法）

### 3.1 五问（必须全部回答才能设计）

```markdown
1. 原生是否已有能力？
2. 是否已有官方扩展点？
3. 社区是否已有成熟方案？
4. 是能力缺失还是不会使用？
5. 如果重新开始，我还会设计这个模块吗？
```

**铁律：没有完成这五问，禁止设计新机制。**

### 3.2 五问执行模板

对每个"我想做 X"的念头，填写：

```yaml
想法: <一句话描述想做的事>
Q1_原生能力: <Cline 是否已有，具体是哪个，引用文档>
Q2_扩展点: <是否有 Hook/MCP/Workflow/Skill 扩展点>
Q3_社区方案: <是否有人做过，链接>
Q4_缺失还是不会用: <二选一 + 理由>
Q5_重新开始还会做吗: <是/否 + 理由>
结论: <放弃 / 用原生 / 用扩展点 / 新建>
```

### 3.3 五问的默认倾向

> **默认答案是"放弃"或"用原生"**。只有五问全部走完仍指向"新建"，才允许新建。

这个默认倾向是对抗"太快相信这个能力不存在所以我要自己实现"的失败模式。

## 4. 长期记忆治理（清理战略债）

### 4.1 记忆四分类

| 类别 | 含义 | 去向 |
|------|------|------|
| **项目事实** | 技术栈、目录结构、配置 | Memory Bank（事实性） |
| **认知原则** | 思考方式铁律 | constitution / 铁律（C 类） |
| **运行时补丁** | shell 经验、PowerShell 处理 | 标记退休的 B 类清单 |
| **战略决断** | 未来方向选择 | **必须做决策，不允许滞留** |

### 4.2 战略债清理规则

```
战略决断
  ≠
待办事项
```

- "未来再研究" → 必须给出**决策截止日期**，否则删除
- "下阶段再处理" → 必须定义"下阶段"的触发条件，否则删除
- "等 benchmark 后决定" → 必须定义 benchmark 的执行计划，否则删除

> **绝不允许战略决断伪装成待办**。这是产生"战略债"的根源。

### 4.3 记忆健康度检查

每周一次，三个指标：

| 指标 | 健康阈值 | 超标动作 |
|------|----------|----------|
| 新增"战略决断"类记忆数 | ≤ 0 | 超标则强制做决策 |
| 新增"运行时补丁"类记忆数 | ≤ 2 | 超标说明在治表不治本，回到 A 类机制化 |
| 新增"认知原则"类记忆数 | 不限 | 这是真正的资产积累 |

对以下每一项，验证"Cline 原生是否已覆盖"，并标注本项目当前是"重复 / 增强 / 真补缺"：

### 5.1 验证清单

| Cline 原生能力 | 本项目对应物 | 关系判定 | 处置 |
|---------------|-------------|----------|------|
| Plan/Act 模式 | Skills 的 plan/act preferred_mode | ? | 待验证 |
| Hooks（6 种） | frontmatter permissions + 铁律 | ? | 待验证 |
| Memory Bank（6 文件） | 三层模型 Memory/Handoff/Compact | ? | 待验证 |
| Checkpoint（影子 Git） | ? | ? | 待验证 |
| Rules（条件触发） | constitution / SKILL.md | ? | 待验证 |
| @Mentions | file-search skill | ? | 待验证 |
| Workflows | executing-plans? | ? | 待验证 |
| Subagents | requesting-code-review | ? | 待验证 |
| Slash Commands | ? | ? | 待验证 |
| CLI headless | ? | ? | 待验证 |
| 自动上下文摘要 | Compact 层 | ? | 待验证 |
| Deep Planning | writing-plans / brainstorming | ? | 待验证 |

### 5.2 验证产出

每项填完后，得到三张表：


1. **compact** —— 本项目的 Compact 层是否就是 Cline 自动摘要
2. **checkpoint** —— 是否需要自己做，还是 Cline 已够用
3. **workflow** —— 本项目 Skills 是否该改为 Cline Workflow
4. **hooks** —— 本项目铁律是否该全部改为 Cline Hook
5. **MCP** —— 本项目 6 工具是否与 Cline MCP 重复

## 6. 本项目的真正价值锚点

### 6.1 三个真补缺方向（基于 Cline 调查）

| 方向 | 为什么是真补缺 | A/B/C 类 |
|------|--------------|----------|
| **Loop Engineering 元语**（目标验证器/无进展检测/Reflexion/预算） | Cline 完全缺失 | A 类（未来可代码化） |
| **Windows Hook 替代** | Cline Hook 仅 macOS/Linux | B 类（Cline 限制） |
| **向量长期记忆 + 动态重规划** | Cline 弱项 | A 类 |

### 6.2 真正长期保留的资产

| 资产 | 为什么长期 | A/B/C 类 |
|------|-----------|----------|
| **铁律体系**（证据优于推测、问题定义优于方案等） | 治理思考方式 | **C 类** |
| **Skills 方法论封装**（TDD/调试/头脑风暴流程） | Cline Skill 是空的，本项目提供预置库 | C 类 |
| **三层模型**（Memory/Handoff/Compact 理念） | 方向正确，Handoff 是 Cline 没有的 | C 类（理念）/ A 类（实现） |

### 6.3 应退休的资产

| 资产 | 退休原因 | 退休方式 |
|------|----------|----------|
| 大量 shell 经验规则 | A 类，应被机制替代 | 移入"待机制化"清单 |
| PowerShell 阻塞处理规则 | A 类 | 等 timeout/watchdog 机制 |
| 重复循环处理规则 | A 类 | 等 loop guard 机制 |

## 7. 开发节奏（强制纪律）

### 7.1 单周节奏

```
周一：明确本周一个功能目标（非治理目标）
周二-周四：执行，每天检查"是否又在写规则"
周五：A/B/C 分类复盘 + 记忆健康度检查
```

### 7.2 禁止事项

- 禁止"顺手写个规则"——必须走五问门控
- 禁止"先记下来以后再说"——战略决断必须当场决策或删除
- 禁止"再优化一下治理"——治理优化不计入本周功能目标
- 禁止"补一个长期记忆"——先过四分类

### 7.3 强制事项

- 每个新功能开门前填五问模板
- 每次写规则前先判定 A/B/C 类
- 每周做一次记忆健康度检查
- 额度用完立即停止并复盘

## 8. 失败模式预警（贴在显眼处）

### 8.1 已识别的失败模式

```
发现问题 → 写规则 → 写经验 → 写记忆 → 写治理 → 再发现问题
```

预警信号：
- 本周新增文件全是 .md（规则/记忆/治理），没有功能代码
- 出现"未来再研究"类待办
- 产生"我到底在做什么"的困惑
- 规则数增长快于功能数

### 8.2 最大失误的根因

```
设计能力 > 调研能力
```

默认路径是"发现缺口 → 自己设计"，正确路径是"发现缺口 → 先验证原生是否已有能力"。

### 8.3 根治方法

> **新功能启动前五问**是唯一根治手段。没有完成五问，禁止设计新机制。

## 9. 决策原则（Decision Principles）

### 9.1 决策分层

```
ADR        记录"选什么"（Decision-level）
Design Doc 记录"怎么做"（Implementation-level）
不混写
```

- **ADR**：1-2 页极薄，只填 Context / Alternatives / Decision / Consequences / Status 五字段
- **Design Doc**：实现细节、接口契约、配置项、降级方案、模板等全部进入
- **判断归属**：如果这个决策被否决后会让 ADR 整体推翻，则属于 Decision-level；如果只是局部参数/策略，则属于 Implementation-level

### 9.2 收敛触发

```
当 ADR 关键字段已填齐时，停止发散，进入执行
```

- 5 个字段都能写出具体内容 = 决策收敛点已到达
- 此后若再抛新问题，按 9.3 处理

### 9.3 收敛阶段禁止

- **禁止引入新的核心问题**——收敛阶段只做综合，不做发散
- **禁止把实现细节塞回 ADR**——会导致 ADR 膨胀成设计文档
- 违反信号：在 "Accepted" 之前还在反复抛 "要不要 X / 要不要 Y" 的细节

### 9.4 ADR 模板（参考）

```
# ADR-NNN: <主题>

- Status: Proposed | Accepted | Deprecated | Superseded
- Date: YYYY-MM-DD

## Context
<事实、约束、决策背景>

## Alternatives
<候选方案列表，每个一行特征>

## Decision
<采用 / 暂缓 / 拒绝，及理由要点>

## Consequences
<正面 / 负面 / 退休条件>

## Next Steps
<ADR 之后的直接后续任务，如 Capability Probe / Design Doc>
```

## 10. 协作原则（Collaboration Principles）

### 10.1 Preferred Collaboration Pattern

```
1. Trae 负责方案展开
2. 用户负责过滤与约束补充
3. 外部评审（GPT 或其他来源）负责提供第二视角
4. Trae 负责最终收敛与落地

默认采用该模式，但不是强制要求。
```

> 这是当前**有效**的协作方式，不是固化流程。未来可能演变为"用户 → Trae 直接收敛" / "用户 → 多个评审源 → Trae" 等形态。

### 10.2 收敛节奏

```
T1  Trae 发散（多方案 + 必要审视，1 回合内完成）
T2  外部评审（GPT 或其他来源）反馈
T3  Trae 综合收敛，输出文档/代码
```

- 经验值：2 轮发散-反馈-收敛通常足够
- 用户的过滤是判断节点，不是传话——加入用户判断后再反馈给 Trae
- T3 阶段严格只做综合，参见 9.3

### 10.3 反模式

| 信号 | 含义 | 处置 |
|------|------|------|
| Trae 在 T3 抛出新问题 | 失控，准备进入第 4 轮发散 | 停下，用框架字段触发收敛 |
| 同一主题超过 3 轮发散仍未输出产物 | 框架未触发或框架选错 | 重新定性问题，换框架 |
| 把协作工具（具体某个模型）写成必经节点 | 流程依赖具体工具，未来难替换 | 改为 Preferred Pattern 表述 |
