# 观察记录：write_to_file 中断 → 无限重试卡死模式

## 现象描述

当 write_to_file 工具调用在**执行过程中被中断**（会话超时、用户手动打断、工具调用中途失败），发生以下连锁反应：

1. **文件未完整写入**：中断时的文件可能不完整或完全不存在（取决于写入阶段），但在 Cline 的上下文中，工具返回状态不可知。
2. **任务恢复后重新读取 → File not found**：Cline 尝试读取刚写的文件以继续任务，发现文件不存在。
3. **进入无出口重试循环**：Cline 认定文件必须存在才能继续，但写操作又涉及调用已经失败过的 write_to_file，且在缺乏外部输入的情况下无法打破自洽循环。

## 本项目发生过的情况

2026-06-28：写 `docs/decisions/D-2026-06-28-cline-v401-sdk-rollback.md` 时被用户中断。恢复任务后：
- `read_file` → "File not found"（之前的写入未生效）
- Cline 尝试重复读/写该文件，陷入判断死循环
- 用户手动"继续"指令才打破循环，最终用 write_to_file 重新写入成功

## 根因

整个链路的根本原因是：

1. **write_to_file 不是原子操作**——它涉及目录创建、文件写入、文件截断等子步骤，任意一步被中断都会导致不一致状态。
2. **Cline 没有"检查文件是否完整"的语义**——文件存在 ≠ 内容完整，但 Cline 的工具使用模式默认文件存在即内容正确。
3. **恢复机制缺少对未完成写入的感知**——Task Resumption 不记录上一次 tool call 的执行状态。Cline 不知道"上一次写文件没写完"。

## 建议的处理方式

如果以后再遇到"文件似乎应该存在但 read_file 找不到"的情况：

1. **直接重写**：不要尝试先 read 再决定——文件不存在说明上次写入未生效，直接走 write_to_file 覆盖即可。
2. **提前打破循环**：如果 Cline 连续 2 次尝试读同一不存在的文件，自动判断为写入中断的遗留状态，改用 ls 检查目录内容确认文件是否存在。
3. **区分意图**：如果文件是"补充性文档"（如决策笔记），不存在不会导致下游失败，应先继续不阻塞在下游。
