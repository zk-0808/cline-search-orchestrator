### **提示词层软过滤无法等效 Goggles 的硬排序，关键不在"排序精度"而在"介入点"**

提示词层让 LLM 按 BOOST/DOWNRANK/DISCARD 表打标，本质上是在搜索引擎已经把候选集裁剪到最终 10–50 条结果之后再做一次**软重排**；而 Brave Goggles 是在**召回阶段**对数万条候选执行确定性硬过滤。二者介入管线的位置不同,决定了软过滤可以近似"对已出现结果的重新加权与丢弃",但**无法恢复被引擎在上游丢掉的域名**。降级损失因此分裂成两个量级完全不同的部分:可重排部分的精度损失约在 NDCG 几个百分点级别(若工程到位甚至接近持平),而召回覆盖面的结构性塌缩则是无法用 NDCG 衡量的、可能高达数量级的损失,同时叠加延迟(约 100–1000×)、成本(near-0 → 按 token 计费)与非确定性三重代价。

下面分层展开。

### **一、为什么"介入点"是决定性变量**

Brave 在 Goggles 白皮书里把这一点讲得非常直白。现代搜索引擎分阶段执行:**召回阶段(recall phase)**用简单特征把数十亿页面砍到"数千条(in the order of few thousands)"的候选集;随后的**精排阶段(precision phase)**用一系列越来越昂贵的模型层层收窄;最后的 ranking 阶段只处理一个很小的候选集并产出给用户的最终顺序。白皮书明确指出:"Goggles 越早集成进搜索流程越有效,这样更多页面才能被规则作用到。"以经典的 **"过滤掉互联网 top 1000 域名"** 这条 Goggle 为例——如果把它应用在最终结果集上,"对大多数查询会导致结果极少、甚至没有(very few results, if any)",因为搜索引擎本身就有把流量导向热门域名的内在偏置。规则必须作用在尽可能大的候选集上,候选与规则的交集才不至于为空。 [Brave](https://brave.com/static-assets/files/goggles.pdf)

白皮书还专门点出了"下游软过滤"为什么不可行:"Bing API 的 top 10 或 top 50 结果远远不够大(not nearly big enough)";即便靠抓取拿到 100 条,"虽然比 10 条好,但仍然太小了"。唯一高效的实现方式,是搜索引擎允许用户把自定义重排函数下发到**第一批结果(通常数以万计)**上,而不是在候选集已经被收窄到所剩无几的最后阶段。 [Brave](https://brave.com/static-assets/files/goggles.pdf)

这正是提示词层软过滤的结构性天花板:LLM 拿到的就是那个"已经被收窄到所剩无几"的结果集。上面那张图刻画的就是这个落差——**被引擎在召回/精排阶段丢弃的域名,永远不会到达 LLM 这一层被重新加权。** DISCARD 一个本来就没出现的域名是无意义的,BOOST 一个从未被召回的长尾优质站点更是不可能的。

### **二、软过滤能等效的部分:对"幸存结果"的重排与丢弃**

公平地说,软过滤并非一无是处。Goggles 的 DSL 语义本身相当简单——API 文档里它只有三个核心动作:`$boost`(强度 1–10)、`$downrank`(强度 1–10)、`$discard`(完全移除),配合 `site=`、路径、通配符等 URL 触发器,以及一套确定性的冲突消解优先级($discard 最高,$boost 压过 $downrank,高强度压过低强度)。 [Brave Search API](https://api-dashboard.search.brave.com/documentation/resources/goggles)

这套"对一个已知列表做加权/降权/剔除"的语义,恰恰是 LLM 重排比较擅长的区间。RankGPT 的实验显示,GPT-4 用零样本指令式排列生成,在 TREC、BEIR、Mr.TyDi 上平均 nDCG@10 达到 53.68,甚至超过 monoT5(3B)、Cohere Rerank-v2 等强监督系统。 [GitHub](https://github.com/sunnweiwei/rankgpt) [DataCamp](https://www.datacamp.com/fr/tutorial/rankgpt-rag-reranking-agent) 换句话说,**如果任务被限定为"在一个已经召回好的候选集上重新排序",精心工程化的 LLM 重排在排序质量上是可以打平甚至略胜传统方法的。** Brave 自己推出的 Rerank 功能(底层就是 Goggles 技术)也印证了这个区间的价值——它让用户对结果页里的域名做 thumbs up/down,被点赞的域名后续上浮、被点踩的不再出现。 [Brave](https://brave.com/blog/search-rerank) 这部分语义,软过滤确实能近似复刻。

### **三、软过滤无法等效的部分:四类损失来源**

问题在于,真正让 Goggles 有价值的能力,大多落在软过滤够不到的地方。

**第一,召回覆盖的塌缩(最大、且无法用 NDCG 衡量)。** 如前所述,"排除 top 1000 域名""扶持小众独立媒体""只搜某地区高质量徒步站点"这类 Goggle 的全部意义,就是把默认排序埋掉的长尾翻出来。这要求在数万条候选上操作。软过滤拿到的 10–50 条几乎全是热门域名,对它们执行 DISCARD 后接近空集。这不是排序误差,而是召回误差——在错误的候选集上算出再漂亮的 NDCG 也无济于事。这一项的损失上不封顶,取决于查询有多依赖长尾。

**第二,非确定性与不可复现。** Goggles 是纯确定性的:相同输入必然产出相同结果,可审计、可用 Git 版本管理(API 文档因此建议"用 Git 跟踪你的 Goggles 文件")。 [Brave Search API](https://api-dashboard.search.brave.com/documentation/resources/goggles) 而 LLM 重排存在著名的**位置偏置(position bias)**与"lost-in-the-middle"现象:列表末尾的文档更难被提到顶部,排序对输入顺序高度敏感。RankVicuna 的作者直接把"基于闭源 API 的重排结果不可复现、非确定性(non-reproducible and non-deterministic)"列为核心动机。 [Hugging Face](https://huggingface.co/papers?q=listwise) 把这种随机性放进一个本应"硬"的域名排序里,等于丢掉了 Goggles 最被看重的可信与可审计属性。

**第三,自然语言指令对"强度"和"偏置"的控制力很弱。** Goggles 的 `$boost=1..10` 是确定性整数权重,冲突消解规则清晰。LLM 的"软"打标则是模糊的。有研究在 MovieLens-100K 上用 GPT-4o 重排 LightGCN 的 top-k,结果在所有 NDCG@k 上都**劣于**基线模型;更关键的是,**显式在提示词里指示 LLM "减少位置偏置"不仅无效,多数情况下还让性能进一步下降。** [arXiv](https://arxiv.org/html/2505.04948v1) 这直接打击了"用提示词指令去精确控制排序行为"的可行性——你写在表里的 DOWNRANK 强度,模型未必照办。即便是专门做去偏的 DebiasFirst,通过 IPS 校准 + 位置感知数据增强(且需要微调,而非纯提示词),也只换来 nDCG@10 平均 **2%–4%** 的提升,可见位置偏置之顽固。 [arXiv](https://arxiv.org/html/2604.03642v1)

**第四,规模上限。** Goggles 单文件支持最多 **10 万条指令、2MB**,在召回阶段对数万 URL × 数千规则匹配仍是毫秒级。 [Brave Search API](https://api-dashboard.search.brave.com/documentation/resources/goggles) LLM 受上下文窗口约束,listwise 排序"常常因 token 限制而不可行",只能滑动窗口分批处理。 [arXiv](https://arxiv.org/html/2505.04948v1) 一张几万行的域名白/黑名单,根本塞不进提示词。

### **五、工程结论:何时降级可接受,何时不可**

把上述拆解收敛成一句可操作的判断:**软过滤等效的是 Goggles 语义的"投影",而非 Goggles 系统效应的全部。**

当你的需求落在"对一个已经足够大的候选集做重排序,或剔除少数已出现的劣质域名"时,提示词层软过滤是一个可接受的降级——排序质量损失可控(NDCG 几个百分点,工程到位可接近持平),代价主要是延迟和成本上升,以及失去确定性。这对应 Brave Rerank 那种"对结果页可见域名做上浮/剔除"的轻量场景。

但当需求本质是**域名级硬排序在全量召回上的应用**——典型如"排除 top N 热门域名以发掘长尾""把小众权威站点系统性提到前面""按数万条域名清单做白/黑名单"——软过滤会发生不可恢复的结构性失败:它够不到那些从未被召回的页面,而这恰恰是这类 Goggle 的全部价值所在。在这种场景下,损失不是"差几个百分点",而是"做的根本不是同一件事"。要真正等效,唯一路径是把过滤逻辑下沉到召回阶段、由持有完整索引的搜索后端执行——这正是 Brave 白皮书反复强调、且承诺由 Brave 搜索引擎在索引层兜底实现的核心架构前提。 [Brave](https://brave.com/static-assets/files/goggles.pdf)

如果这是你 P1 研究里要写进结论的一条,我建议的措辞是:**软过滤可视为 Goggles 在"最终结果重排"子空间上的有损近似;其精度损失有界(NDCG 级),但召回损失无界(取决于查询对长尾的依赖度),且必然牺牲确定性、延迟与成本三项硬指标。** 需要我进一步把这套对比沉淀成一份可下载的研究备忘录(含图表与引用),或针对某一类具体查询做召回覆盖的定量模拟,可以继续。




2. P4 Same-Source Merge

I'll pull the two most authoritative primary sources (Manning IR Book and the Google/Manku SimHash paper) for precise grounding before analyzing the equivalence question.### **命题前半段完全成立；"等价"则需分场景——逐字镜像下提示词层与现成方法目标等价但被工程性严格压制，唯有"语义级同源"才是现成指纹法做不到、提示词层可能真正不等价的地方。**

同源转载/镜像去重不是模糊地带,而是有教科书定义、有 WWW/SIGIR 经典论文、在工业搜索引擎里跑了二十年的成熟问题。但"项目提示词层实现是否与现成方法等价"这个判断,取决于去重对象到底是**逐字/近逐字的镜像**,还是**语义改写的转载**——两者的答案截然相反。

### **一、命题前半段:这确实是 IR 的"已解决"问题**

Manning 的《Introduction to Information Retrieval》在第 19.6 节"Near-duplicates and shingling"里把它当作 Web 搜索的基础挑战来讲:据估计 Web 上约 40% 的页面是其他页面的副本,其中很多是"合法副本"——比如信息库为了冗余和可用性而做的镜像(mirroring),搜索引擎要避免重复索引以压低存储和处理开销。这正是"同源转载/镜像"的标准动机表述。教科书给出的标准管线是:对完全相同的页面用 64 位指纹(fingerprint)直接去重;对"近重复"(只差广告、计数器、最后修改时间戳等)用 **shingling**(k-shingle,典型 k=4)把文档表示成 k-gram 集合,用 **Jaccard 系数**(阈值如 0.9)衡量重叠,再用 **MinHash**(随机置换取最小值,200 个置换构成 sketch)把两两比较降到可估计,最后用 **union-find** 聚成"语法簇"(syntactic clusters)。[Stanford NLP Group](https://nlp.stanford.edu/IR-book/html/htmledition/near-duplicates-and-shingling-1.html)

工业侧的代表是 Manku 等人(Google)在 WWW 2007 的"Detecting Near-Duplicates for Web Crawling":完全重复(镜像、抄袭)用校验和即可识别;近重复用 **Charikar 的 SimHash** 把文档向量降维成 64 位指纹,near-duplicate 的指纹只差很少的比特(实验取 k=3,精确率/召回率均约 0.75),并配套解决"汉明距离问题"以在 80 亿页规模上毫秒级查询。该文第 5 节是一份系统性 survey,明确把"识别 Web 镜像(web mirrors)"列为去重的首要目标,并引用 Bharat & Broder 的"Mirror, mirror on the Web"作为镜像识别的专门方法。[Google Research](https://research.google.com/pubs/archive/33026.pdf)

换言之,从"目标定义—特征抽取—签名压缩—大规模检索"整条链路,镜像/转载去重都有现成、可证、被反复验证的方案。**项目若声称这是新东西,这一点站不住脚。**

### **二、"等价"的关键分水岭:syntactic 还是 semantic**

现成方法(shingling/SimHash)做的是**句法相似度**(syntactic similarity),而非语义。教科书和讲义都把这点说得很直白:真正的"语义相似"(同一内容用不同措辞表达)"太难计算",所以工程上只做 token/k-gram 重叠这种句法判断。Manku 的 survey 也专门把"给定一篇新闻、找其他来源报道同一事件"归为**另一类问题**(semantic 而非 syntactic),其解法是 LSI、链接结构(Dean & Henzinger 的 related-pages),而**不是** shingling/SimHash。[Google Research](https://research.google.com/pubs/archive/33026.pdf)

这正是判断"等价性"的分水岭:

- **逐字/近逐字镜像(改广告、时间戳、站点 chrome)**:这是 SimHash/shingling 的主场。提示词层的 LLM 实现与现成方法在**目标上等价**,但在工程属性上被**严格压制**——它更贵、更慢、不确定、阈值不可证、无法扩到语料级。这里没有任何算法新意,用 LLM 反而是杀鸡用牛刀。
- **语义级同源(改写、洗稿、翻译、摘要式转载、跨语言搬运)**:句法指纹被文献明确记录为"做不好"——换词就会让相似文档的 shingle 集/指纹拉开。这恰恰是提示词层 LLM **可能真正不等价(更强)**的地方,因为它能基于语义判断"这两篇是同一来源的不同包装"。**但这已经不是经典意义上的"镜像去重",而是一个更难、定义更模糊、需要单独验证的问题。**

所以:**等价与否,完全取决于项目"同源合并"瞄准的是哪一类。** 如果瞄准的是逐字镜像,那它与现成方法等价(且劣化);如果瞄准语义转载,那它不等价,但也不能再借"成熟 IR 技术"的成熟度为自己背书——因为成熟的恰恰是它没在做的那部分。

### **三、一个常被忽略的工程错位:语料级 vs 结果级**

现成方法那套重型机械(MinHash sketch、super-shingle、汉明多索引探测、MapReduce 批处理)是为**索引期、语料级、十亿量级**的去重设计的,核心约束是"避免对数十亿文档做平方级两两比较"。[Stanford NLP Group](https://nlp.stanford.edu/IR-book/html/htmledition/near-duplicates-and-shingling-1.html)

但提示词层的"同源合并"通常发生在**查询期、对一份检索结果列表(几条到上百条)**做去重。在这个尺度上:

- 平方级 blowup 的约束根本不存在,连 MinHash/super-shingle 都用不上;
- 真正"现成、对标"的方案是**朴素的两两 SimHash/Jaccard + URL 规范化(canonicalization)**——几行代码、确定性、零模型成本就能拿下逐字镜像。

这反而**加强**了"逐字场景下提示词层是 overkill"的批评:它不是在和 Google 的重型基础设施竞争,而是在和几行指纹代码 + URL 归一化竞争,后者更便宜、更可控。提示词层唯一站得住的差异化,仍然只剩"语义合并"和"它本来就在回路里、零额外基础设施"这两点。

### **四、"同源"本身有歧义:同站 vs 跨站(实证证据)**

还有一个能让结论更锋利的实证点。Henzinger 在 1.6B 页面上的大规模评测发现:shingling 和 SimHash 对**不同站点之间**的近重复对都有高精确率,但对**同一站点内部**的近重复对都表现很差。[Penn State](https://clgiles.ist.psu.edu/pubs/DOCENG2013-near-duplicate-detection.pdf)

把这个发现映射到"同源":

- 若"同源转载/镜像"指**跨站搬运**(A 站文章被 B、C、D 站逐字转载),那正落在现成方法的强项区,提示词层对逐字内容毫无增量;
- 若指**同站近重复**(URL 变体、faceted 导航、分页),现成 syntactic 指纹本就弱——但此处的正解是 URL 规范化/结构信号,通用 LLM 提示同样**不是**天然更优的解法。

无论哪种解读,"用提示词层做逐字同源去重"都很难论证出相对现成方法的优势。

### **五、维度对照与裁决**

| 对比维度 | 现成 IR 方法(SimHash / shingling+MinHash) | 提示词层(LLM)实现 |
| --- | --- | --- |
| 去重目标 | 折叠重复内容、降索引/存储成本、提升 top-k | 同一目标(逐字场景下完全等价) |
| 相似度类型 | 句法(token/k-gram 重叠、指纹汉明距) | 可达语义(改写/翻译/摘要)——唯一真正差异点 |
| 理论保证/可调性 | 有(MinHash 无偏估计 Jaccard;阈值显式可证、可调) | 无形式保证、阈值隐式、输出不确定 |
| 规模/成本/延迟 | 十亿级、毫秒级、近零边际成本、可离线批处理 | 受上下文窗口限制、每次调用有成本与延迟 |
| 作用环节 | 索引期、语料级 | 查询期、结果列表级(几行指纹代码即可平替) |

**裁决**:核心命题的前半段(成熟 IR 技术)无可争议;后半段的"等价"判断应当是——**在逐字/近逐字镜像这一经典定义下,提示词层实现与现成方法目标等价、工程上被严格压制,且在结果级尺度下属于明显的过度工程,没有算法新意。** 它唯一能主张"不等价/有价值"的窗口,是**语义级同源合并**;但一旦走进这个窗口,项目就必须用语义任务自己的评测(精确率/召回率、对洗稿与翻译的鲁棒性、误并风险)来证明,而**不能**再援引 shingling/SimHash 的成熟度——因为那套成熟度恰恰不覆盖语义这部分。因此,要让"提示词层同源合并"摆脱"重造轮子"的指控,需要明确做到两件事:其一,声明对象是语义转载而非逐字镜像;其二,给出相对"SimHash/Jaccard + URL 规范化"基线的、可复现的语义去重增益证据。否则按现有命题,判定为**与现成方法等价(且在逐字场景下劣于现成方法)**是成立的。




3. #20 反证检索

---

### **一、支持命题的核心证据：负向召回差发生在"检索之前"，是结构性的**

最直接的证据来自专门测否定的 IR 基准 **NevIR**。该研究众包标注了"两段几乎相同、仅靠一处否定区分"的文档对，结果发现绝大多数神经检索模型在否定上的表现**等于或低于随机排序**：bi-encoder、sparse、late-interaction(ColBERT) 全部低于随机，只有 cross-encoder 略高于随机。论文的结论非常关键——"first-stage retriever 在检索时根本没有把否定纳入考虑；要在否定上做对,昂贵的 cross-encoder 重排是必要但仍不充分的,而且模型必须把否定写进训练数据"。换句话说，召回失败发生在第一阶段匹配,而提示词作用于生成端,够不到那一层。 [ACL Anthology](https://aclanthology.org/2024.eacl-long.139.pdf)

复现与扩展研究（SIGIR 2025 reproducibility）进一步确认：即便换上最新模型，多数神经 IR 仍在否定上挣扎，新出现的 **listwise LLM reranker** 表现最好,但**仍低于人类水平**。趋势在改善,瓶颈未消除。 [arXiv](https://arxiv.org/html/2502.13506v1)

机理层面的"实锤"来自 TREC 2025 BioGen 的论文 *Negation is Not Semantic*：作者把对抗式稠密检索用于反例（矛盾证据）检索时，出现所谓 **"语义坍缩"(Semantic Collapse)——否定信号在向量空间里根本不可分**，对抗式稠密方案的 MRR 只有 **0.023**（近乎全错）。其根因解释是：稠密 embedding 把"同一主题"误当成"同一立场",而否定/反例正是"主题相同、立场相反",于是被几何性地抹平。 [arXiv](https://arxiv.org/html/2603.17580)

多模态侧的 **NegBench/MIT** 研究给出了对"提示词可治"最致命的一击：视觉-语言模型存在 **"肯定偏置"(affirmation bias)**，带否定的图像检索召回直接掉约 **25%**,且研究者明确指出——**"不管你怎么表达否定或排除,模型都会直接忽略它",这一点在所有被测模型上一致成立**。"换个说法/改写 query"恰恰就是提示词层面的操作，而证据显示它无效。配套的负向理解综述还发现一个反直觉规律：**模型越大,对否定反而越不敏感**。 [MIT News](https://news.mit.edu/2025/study-shows-vision-language-models-cant-handle-negation-words-queries-0514) [arXiv](https://arxiv.org/html/2502.09645v1)

---

### **二、对 DDG / 通用引擎的特异性证据：算子在产品层被"砍掉"了**

命题点名 DDG 是有依据的。DuckDuckGo 本身**基于微软 Bing**，且自 2023 年起把大部分搜索算子（精确匹配 `""`、`+`、`-` 权重、`filetype:` 等）基本下线，一度只剩 `site:` 可靠；官方随后澄清"算子仍在,但承认它们'不是一直都能工作'"。也就是说，对 DDG 而言，"后端能力限制"几乎是字面意义上的真实——负向/精确算子在产品层被移除或变得不稳定，这不是任何 query 措辞能绕过的。同文也指出 Google 同样不再稳定遵守 `""` 等算子。 [gHacks](https://www.ghacks.net/2023/04/24/duckduckgo-disables-most-search-filters-from-search)

---

### **三、反证与修正：命题在三个地方"过强"，需要收紧**

这是本次反证检索的重点——以下证据与"一刀切的后端无能、且不可治"相左：

**反证 1：这不是"一个"限制，而是三种机制，命题把它们混为一谈。**

| 机制 | 典型 query | 失败发生层 | 能否靠"改写提示词"治 | 真正有效的修法 |
|---|---|---|---|---|
| 布尔 OR（析取扩召回） | `A OR B` | 产品/算子支持层 | 否（引擎可能忽略 OR 退化为 AND） | agent 侧拆成多次检索再取并集 |
| 否定词 NOT/排除 | `X -Y`、`无Y的X` | 词项级 + 语义级 | 部分（见反证 2） | 词法精确匹配 / 后置过滤 |
| 反例/反证词 | "驳斥/无关联/no effect" | 架构级（entailment trap） | 否 | 高召回词法 + NLI/reranker |

**反证 2：纯词法后端其实"吃得下"显式否定线索，问题更多在 query 构造而非后端无能。** BioGen 论文明确指出，否定标记如 *"absence of"、"no evidence of"、"no signs of"* 这类线索,**BM25 的精确匹配反而比稠密 embedding 抓得更好**；其最终最优系统正是放弃稠密、改用"解耦的词法(BM25)+NegEx 否定过滤+句级 NLI"。这意味着：把"否定/排除"改写成**正向词项线索**去检索，在词法引擎上是真能提升召回的——这部分更像"检索策略/查询构造"问题，而不是后端绝对无能。 [arXiv](https://arxiv.org/html/2603.17580)

**反证 3："可治",只是不靠 prompt。** 所有被验证有效的修法都落在后端/数据/训练，而非措辞：NegBench 用百万级否定样本微调 CLIP，否定 query 召回 **+10%**、否定多选准确率 **+40%**；亚马逊用训练出的 Seq2Seq 做**否定 query 改写**，P@30 提升 **3.17%**（幅度小但为正）；指令微调(instruction tuning)也被证明能改善 LLM 的否定理解。这些恰恰从反面印证命题主旨——**治本要换架构/数据，而不是换提示词**；同时也否定了"不可治"的绝对说法。 [NegBench](https://negbench.github.io) [Amazon Science](https://www.amazon.science/publications/learning-to-rewrite-negation-queries-in-product-search)

### **四、校准后的结论与对 Agent 的可操作启示**

把证据合起来看，命题应当被这样精确化：

> **在"同一后端 + 同一 query"前提下，靠堆叠或改写提示词无法挽救负向 query 的召回——这一点成立且有强证据。** 但"后端能力限制"实为三种机制(OR 算子支持、词项级否定、语义级反例)的合称；其中词法后端对显式否定线索并非无能,且整体问题可由检索架构、训练数据与 agent 侧检索策略缓解。因此"非提示词可治"应改述为**"非（对同一后端的）措辞改写可治,但可由检索策略+架构+训练数据缓解"**。

据此，agent 在实战中遇到负向 query 时的正确动作不是"把否定词塞进一条 query"，而是：遇到 **OR** 就拆成多条检索再取并集；遇到**否定/排除**就转成正向词项检索或做结果后置过滤；遇到**反例/反证**就用"高召回词法检索 + NLI/重排"而非寄望于稠密向量。这与本条目"反证检索"的方法论本身是一致的——反例要靠策略主动去捞，而不是指望后端自动理解你的否定。






4. #21 多样性排序

I have strong initial coverage. Let me fetch the most relevant primary sources to extract concrete numbers that directly support the magnitude/reliability argument.I have rich primary-source evidence. Let me load design guidance for a compact visual that makes the magnitude/noise-floor argument concrete.### **在提示词层让 LLM 做加权算分（DiversityPenalty ±2 叠加 SourceWeight ±10）确实不可靠：LLM 的算术与打分误差本身就在 ±2 量级，足以把多样性信号淹没。**

你这条命题在工程直觉上是对的，而且能被三类实证文献从不同角度支撑起来：**LLM 连基础算术都做不准**、**LLM 给出的数值分会被"压缩"且随提示扰动漂移**、以及**基于提示词的逐条打分排序明显不如比较式/算法式排序稳定**。下面把证据和机理拆开讲，最后给出"算分到底该放在哪一层"的结论。

**一、机理根因：LLM 把数字当 token，而不是连续量**

最直接的命题——"LLM 做数值算分不可靠"——在 NumericBench 这类专门针对基础数值能力的基准里被钉死了。该工作系统评测了 GPT-4o、DeepSeek-V3、LLaMA-3.1-405B、Qwen2.5-72B 等十余个主流模型的六项基础数值能力（识别、算术、检索、比较、求和、逻辑），结论是：即便是只涉及两个数的简单加减，模型都达不到 100% 正确率；随着位数增加准确率持续下降；乘法的准确率"极低"。在 8 选 1（随机基线 12.5%）的设定下，GPT-4o 在数字列表求和（Summary）一项只有 11.6%，比随机还差，多数模型的逻辑序列预测全部低于 20%。论文进一步把根因归结为五点：分词器会把一个数字切成多个 token、训练语料对 `*` 等符号语义不友好、**next-token 预测范式与算术进位逻辑相反（模型先吐最高位再吐低位）**、位置编码对数字缺乏不变性、以及 Transformer 本身的算力上界。这意味着"让模型在脑内把 SourceWeight×权重再减去 DiversityPenalty 求和"这种操作，是踩在它最薄弱的能力上。 [arXiv](https://arxiv.org/html/2502.11075v1)

这一判断在更大范围的失败模式综述里也得到呼应：把碎片化的 LLM 推理失败统一归类的综述（ICML 2025）指出，算术类失败源于"启发式驱动的推理策略"和"有限的数值精度"，本质上是模式匹配而非显式算法执行，因此现实里越来越依赖外部工具调用来补救。 [OpenReview](https://openreview.net/forum?id=vnX1WHMNmz) 而一项专门拆分"计算错误 vs 推理错误"的 ACL 2025 工作发现，**计算错误本身就是数学题做错的主要来源**，且数值越大错误越多——也就是说，当你的打分公式里出现 ±10 这种较大量级与 ±2 的小量级混合相减时，恰恰是模型最容易算错的区间。 [ACL Anthology](https://aclanthology.org/2025.insights-1.16.pdf)

值得注意的是，连最经典的能力地图（Stanford HELM 上的 MMLU）也早就显示：模型在涉及计算的数学/物理子项上表现明显弱于知识类子项，且早期 GPT-3 在数学子项接近随机水平——"会答题"和"会算数"从来不是一回事。 [Stanford HELM](https://crfm.stanford.edu/helm/mmlu/v1.4.0)

**二、量级根因：LLM 的数值分会被"压缩"，±2 的惩罚低于噪声地板**

你命题里最关键的"量级压不过"这点，有一份非常对症的证据。一项测量 LLM 自评打分偏差的研究发现，**LLM 的评分误差存在约 4 倍的方差压缩**（LLM 误差 σ²≈0.21，而语义相似度映射基线 σ²≈0.87），并且这种压缩在跨模型评分（GPT-4o 评 Claude 生成的文本）时几乎一模一样（σ²≈0.23），说明它是 LLM 打分的**普遍属性**，而非个例。压缩意味着模型倾向把分数挤在一个很窄的区间里、绕着一个小的负偏移聚集——在这种机制下，你想用 ±2 去精细区分条目，会被模型自身的量化粒度直接抹平。 [arXiv](https://arxiv.org/html/2602.13862v2)

更糟的是，这些分数还不稳定、不锚定。LLM-as-a-Judge 打分偏差的系统研究证明：仅仅扰动打分提示里的**评分档位顺序、分数 ID 形式（阿拉伯数字/字母/罗马数字）、参考答案分值**，所有被测模型（含 GPT-4o、DeepSeek-V3、Qwen3）的打分都会漂移；而且每个模型有自己的"打分倾向"——无偏条件下 DeepSeek-V3 会给超过一半样本打 5 分、GPT-4o 偏好 4 分、Qwen3 偏好 5 分。最致命的一条发现是：当提示里给的参考答案分值是 1~4 时，**模型倾向于把分数贴着参考分值给**——也就是说分数会被提示里的锚点拽走，而不是稳定反映真实质量。一个会因为"档位换成罗马数字"就抖动、会被参考分值带跑的打分器，去执行 ±2 的细粒度惩罚，信噪比是不够的。 [arXiv](https://arxiv.org/html/2506.22316v2)

下面这张图把"量级关系"直观化：±2 的多样性惩罚不仅比 ±10 的来源权重小一个量级，而且基本落在 LLM 自身打分噪声的带宽之内。**三、排序专属证据：提示式逐条打分（pointwise scoring）本就是最不稳、最弱的排序范式**

把视角收回到"排序"这个具体任务上，证据更加贴合你的场景。一项针对零样本 LLM 排序器的大规模系统研究（1,248 个提示变体、跨 FlanT5/Mistral/Llama3、超过 12,400 GPU 小时）发现：**pointwise（逐条打分）方法在所有排序家族里效果方差最大、最不稳定**，部分提示变体直接产出很差的结果；而比较式的 pairwise、setwise 则稳定得多。更刺眼的是，他们发现"措辞、语气词、角色扮演"这类**与排序算法无关的提示噪声，有时对排序效果的影响比算法本身还大**。换句话说，当你把排序逻辑塞进提示词、靠模型吐数值分来排序时，你得到的名次里混进了大量来自提示表述的伪信号。 [arXiv](https://arxiv.org/html/2406.14117v1)

这与 LLM-as-a-Judge 领域的共识一致：研究反复指出"逐条单独打分"会违反传递性（A>B、B>C 却 C>A），同一个判官在"单独打分"与"两两比较"两种模式下会给出互相矛盾的结果，因此实践上更推荐**比较/排序式**而非**绝对打分式**的评估。RankPrompt 这类工作也用实验证明：把"逐条直接打分（Direct Scoring）"换成"逐步两两比较"，与人类偏好的一致率能提升约 10 个百分点（74.33% vs 直接打分落后约 10%）。 [LangChain](https://www.langchain.com/resources/llm-as-a-judge) [arXiv](https://arxiv.org/html/2403.12373v1)

**四、把证据折回你的设计：算分必须出 LLM、进算法层**

综合上面三层证据，你那条核心命题可以被强化为一个更精确的工程判断：**问题不只是"LLM 会偶尔算错 ±2"，而是 LLM 在提示层做加权算分时，存在系统性的量化压缩、提示敏感漂移、以及逐条打分固有的不稳定，这三者叠加后的噪声地板就在 ±2~±3 量级，恰好等于甚至超过你想施加的 DiversityPenalty。** 于是 ±2 这个本应起决定作用的小信号，被来源权重 ±10 的大项主导、又被自身噪声吞掉，最终多样性惩罚近似失效。

正确的拆分是让 LLM 只做它擅长的、低精度要求的**语义判断**，把数值合成交给确定性代码：

| 职责 | 交给谁 | 理由 |
|---|---|---|
| 判断两条内容是否"题材/来源重复" | LLM（输出离散标签或布尔，不输出分数） | 语义相似性是 LLM 强项，且不依赖算术精度 |
| SourceWeight、DiversityPenalty 的数值合成与相减 | 算法层（普通代码/打分函数） | 加权求和需要精确、可复现、可调参，正是 LLM 的弱区 |
| 最终排序 | 算法层排序，或 LLM 只做 pairwise 兜底 | pointwise 打分排序方差最大，应避免 |

具体到落地，有三个经过文献验证的可操作要点。其一，**让 LLM 产出结构化的"特征/标签"而非"分数"**——例如让它判断"这条与已选内容是否同源/同主题（是/否）",由代码据此减去固定的 DiversityPenalty，惩罚量级完全由你在代码里掌控，不再被 LLM 的量化粒度抹平。其二，**若一定要 LLM 参与排序，用比较式而非逐条打分**——pairwise/setwise 的稳定性和与人类偏好的一致性都显著优于 pointwise direct scoring。其三，**若暂时无法改架构、必须在提示里打分，至少要做去偏加固**：固定档位顺序、提供满分参考答案锚点、多次采样后取多数票或平均，这些都是上述偏差研究给出的、能实测改善稳定性的手段——但它们只是"止血"，并不能改变"数值合成应当离开提示层"的根本结论。 [arXiv](https://arxiv.org/html/2506.22316v2)

一句话收口：你的直觉判断方向完全正确，而且比"LLM 算术不准"更深一层——真正的杀手是**量级失配 + 打分压缩 + 提示漂移**的三重叠加。把 DiversityPenalty 这种需要精确量级控制的逻辑留在算法层，让 LLM 退回到"贴语义标签"的角色，是文献一致指向的工程解法。

### **为什么「真实浏览器」是必要条件——但只覆盖检测栈的前三层**

Cloudflare 的拦截不是单点判断,而是一个分层叠加的检测栈,任何一层不过都会触发 block 或 challenge。理解这一点,才能看清 headless 浏览器的能力边界到底在哪。

上面的图把五层画了出来。真实浏览器内核在**第 1 层(TLS/JA3·JA4 指纹)**天然占优:因为 Playwright/Selenium 驱动的是真实 Chromium 网络栈,握手特征与真 Chrome 一致,而 `requests`、`axios`、默认 `curl` 这类 raw HTTP 客户端单是一次握手抓包就会被识别为非浏览器软件。**第 3 层(JS Challenge)**同理——Cloudflare 注入的是需要真实 JS 执行的加密 proof-of-work 脚本,Playwright/Puppeteer 在真实 Chromium 里把它跑完、保持 session,这正是 raw HTTP 永远做不到的事。这就是命题里「对」的内核。 [Browserless](https://www.browserless.io/blog/how-to-bypass-cloudflare-scraping)

但问题出在**第 2 层(JS API 探测)**:headless Chromium 默认会泄露一堆自动化信号——`navigator.webdriver` 返回 `true`、User-Agent 带 `HeadlessChrome` 标记、`navigator.plugins` 为空、WebGL 与 codec 指纹异常。任何像样的反爬脚本在毫秒级就能抓到。换句话说,**裸跑的 headless 浏览器不是「对路径」,而是「送人头」**。 [Scrapfly](https://scrapfly.io/blog/posts/playwright-stealth-bypass-bot-detection)

### **stealth 插件能补第 2 层,但补不到第 4、5 层**

要让 headless 浏览器活下来,业界标准做法是叠加 stealth 补丁:Python 用 `playwright-stealth`(v2.x,context-manager API,维护活跃),Node.js 用 `playwright-extra` + `puppeteer-extra-plugin-stealth`(共用同一批约 14 个核心 evasion 模块,但 Node 侧自 2023 年后基本停更)。它们修补的就是上面第 2 层那些信号:把 `navigator.webdriver` 改成 `false`、伪造 `window.chrome`、补齐 plugins/languages、对齐 WebGL 与 User-Agent。 [Brightdata](https://brightdata.com/blog/how-tos/avoid-bot-detection-with-playwright-stealth)

但几乎所有技术来源都给出同一个警告:**stealth 只解决指纹层,解决不了基础设施层**。具体说,它管不了:

- **第 4 层 IP 信誉**——数据中心 IP、VPN 段、被标记的子网,在你的浏览器 JS 还没开始跑之前就被网络层拦掉了,再完美的指纹补丁也改变不了流量来源,必须配住宅代理。
- **第 5 层行为分析**——鼠标轨迹、滚动节奏、点击时序。一个「加载页面→固定等待→抓数据→离开」的脚本,其统计特征与真人有显著差异,stealth 对此完全无能为力,需要手写拟人化(随机化击键延迟、贝塞尔曲线移动鼠标等)。
- **TLS 的版本错配**——一个容易被忽略的坑:Playwright 自带的 Chromium 可能落后于稳定版 Chrome 若干个版本,导致 JA4 指纹对不上最新签名;实务上要用 `channel="chrome"` 调用系统 Chrome 来抹平这个差异。 [ByteTunnels](https://bytetunnels.com/posts/playwright-vs-selenium-stealth-which-evades-detection-better)

所以现实战绩是这样的:在 Cloudflare **标准强度**下,Playwright/Selenium 加 stealth「通常能过」;但在 Cloudflare **激进配置**、DataDome、PerimeterX 高安全模式下,两者「经常被拦,偶尔能过」。这恰恰证伪了「单靠 headless 浏览器就是对路径」的说法——它顶多是多层方案里的一层。

### **「唯一」为什么不成立:三个方向的反例**

命题里「唯一」这个词是最经不起推敲的。研究材料至少给出三类反例。

**其一,旁路方案在部分场景仍然有效。** 对于**老式 JS Challenge 和静态检查**,`cloudscraper` 这类只模拟一小部分浏览器行为的库有时能直接过,根本不启动完整浏览器;`FlareSolverr` 则是用一个 headless 浏览器跑完挑战、把 `cf_clearance` cookie 吐给其他 HTTP 客户端复用。它们的软肋是依赖社区更新、对新版挑战和 Turnstile 力不从心,但「有时能用」本身就否定了「唯一」。 [GitHub](https://github.com/luminati-io/bypass-cloudflare)

**其二,有比 Playwright 更隐蔽的同类工具,反而更「对」。** 这是对命题最有力的修正:Playwright 这类工具的根本问题是「浏览器知道自己正在被自动化」。新一代专用工具直接消除了这个前提——`nodriver` 通过裸 CDP(Chrome DevTools Protocol)直连 Chrome,没有 WebDriver、没有 ChromeDriver 二进制、不注入任何变量;`Camoufox` 则在 C++ 引擎层改造 Firefox,使得检测脚本根本找不到「JS 补丁与引擎行为不一致」的破绽。在面对激进反爬时,这类引擎级方案的隐蔽性优于打补丁的 Playwright/Selenium。也就是说,即便限定在「真实浏览器」范畴内,Playwright 也不是唯一、甚至不是最优解。 [ByteTunnels](https://bytetunnels.com/posts/playwright-vs-selenium-stealth-which-evades-detection-better)

**其三,生产级实践正在「去 DIY 化」。** 当本地 Playwright 触顶,主流做法是转向托管云浏览器(Scrapfly Cloud Browser、Browserless BQL、Bright Data Scraping Browser 等)。它们把指纹轮换、住宅代理、TLS 匹配、挑战求解打包进一次 API 调用——比如 Browserless 的 `solve(type: cloudflare)`、Scrapfly 的 `asp=true`。关键在于,你仍可以用 CDP 把本地 Playwright 连上去,但「穿透」的真正工作发生在托管基础设施里,而非你那个 headless 进程本身。 [Scrapfly](https://scrapfly.io/blog/posts/playwright-stealth-bypass-bot-detection)

### **Turnstile 是另一道坎:多数情况下需要 solver**

你单列了 Turnstile 这个搜索方向,值得单独说明,因为它进一步削弱了「浏览器即方案」的论断。Turnstile 属于 CAPTCHA 类挑战,绕过它有两条路:要么靠足够干净的 stealth 指纹让它**自动放行**(non-interactive 模式),要么在触发交互式挑战时**调用 solver**(2Captcha、CapSolver、CapMonster 等)拿到 token 再注入页面。Playwright stealth 能降低触发概率,但「显著降低」不等于「可靠通过」——可靠性来自 solver 或托管服务,而不是 headless 浏览器自身。 [Capsolver](https://www.capsolver.com/blog/Cloudflare/playwright-stealth)

### **对命题的最终收敛**

如果要把你的核心命题改写成一个站得住的版本,我会这样表述:

> 真实浏览器内核(Playwright/headless Chromium 是其中一种实现)是穿透 Cloudflare JS Challenge 与 Turnstile 的**必要执行环境**,因为只有它能跑完加密 JS 挑战并提供天然的 TLS/JS 指纹;但它**既非唯一**(老式挑战有旁路、引擎级工具如 nodriver/Camoufox 更隐蔽、托管云浏览器另成一路),**也不充分**——裸跑必被识别,真正决定成败的是叠加在浏览器之外的 stealth 补丁、住宅代理、拟人化行为与 CAPTCHA 求解。

一句话:**headless 浏览器是「地基」,不是「整栋楼」。** 把它当成多层防御里的一层来设计,而不是当成银弹,才是 2025–2026 年反爬对抗里真正「对」的路径认知。各家来源也一致强调,任何单一技术都会随 Cloudflare 的更新周期性失效,稳定的做法是「住宅代理 + 真实浏览器 + solver」多层叠加,让单层失效不至于让整个爬虫崩盘。 [BrowserStack](https://www.browserstack.com/guide/playwright-cloudflare)
